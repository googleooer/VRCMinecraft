# Baseline — 2026-06-19 (pre-merge, Phase 0)

Source: ClientSim in-editor, captured during this session (load phase — the 8192-chunk
world streams at ~10 chunks/s so steady-state is effectively never reached; these
load-phase numbers are the representative "all the time" state per the user).
Unity 2022.3.22f1, instance `VRCMinecraft@75cf5c41`.

## Key metrics Phase 1 must beat

| Metric | Baseline | Phase-1 target |
| --- | --- | --- |
| Unity profiler `UdonBehaviour.ManagedUpdate` self | **58.6 ms (≈17 fps), 91.8% of frame** | lower (cross-VM tax removed) |
| `GC.Alloc` calls/frame | ~10,002 (~283 KB) | unchanged here (mesh-emit boxing = Phase ③) |
| Coordinator avg cycle | **44.4–45.5 ms** | folded into McWorld; total reduced |
| — `update workers` | 8.4–18.7 ms | ~0 cross-VM overhead component removed |
| — `assign work` | 26.8–35.9 ms | reduced (picker/state-machine now in-VM) |
| — `siblingStep` (within assign) | 22.8–32.1 ms | unchanged work, but no cross-VM re-entry |
| **Cross-VM call proxy** | the `update workers` + `assign work` ms above are the proxy | both should drop after merge |

> No dedicated cross-VM counter was added (McCoordinator is deleted in Task 9). The
> proxy metric is the existing `Coordinator:` cycle log: after the merge, that work
> moves into `McWorld.Update()` and the per-call VM-transition overhead vanishes. Read
> the post-merge delta from the overall `Update: avg … ms` and the absence of a separate
> `McCoordinator.Update()` invoke in the profiler.
