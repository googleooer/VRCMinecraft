# VRCMinecraft

A recreation of Minecraft Beta 1.7.3 running inside a VRChat world, built in UdonSharp with most of the heavy math offloaded to GPU fragment shaders. The goal is parity, not homage: same worldgen from the same seed, same lighting model, same fluid rules, same sky, same physics constants, verified line-by-line against the deobfuscated b1.7.3 Java source.

The whole simulation runs client-side in a VRChat world on a scripting VM (Udon) that is roughly 100x slower than native C#, has no threads, no coroutines, no compute shaders, and no exceptions. Nearly every architectural decision in this repo follows from those constraints.

## Quick start

1. Open the project with **Unity 2022.3.22f1** (see `ProjectSettings/ProjectVersion.txt`).
2. Open the one real scene: `Assets/VRCMinecraft/scenes/Minecraft.unity`. (`Assets/Scenes/VRCDefaultWorldScene.unity` is the untouched SDK default.)
3. Press Play. ClientSim (bundled with the VRChat Worlds SDK) spawns a local player; walk around and break or place blocks with the 9-slot palette in `ModifyTerrain`.
4. Device builds go through the VRChat SDK control panel (Build & Test). Quest is selected purely by switching the Unity build target to Android: Udon has no runtime platform query, so all PC-vs-Quest behavior is compile-time `#if UNITY_ANDROID`.

Dependencies (resolved via VPM/UPM, see `Packages/vpm-manifest.json` and `Packages/manifest.json`): VRChat Worlds SDK 3.10.2, standalone UdonSharp 1.2.0 (`com.merlin.UdonSharp`, not the SDK-bundled fork), BlueAmulet's UdonSharpOptimizer 1.0.12, NUMovement 0.4.2 (character controller base), VRRefAssist (the `[Singleton]` / auto-reference attributes that wire the scene together at build time), HierarchyDecorator, and MCPForUnity (an editor bridge used with AI assistance to speed up boilerplate and iteration).

`Assets/csc.rsp` defines the `LOGGING` symbol. The verbose perf summaries and debug logs it gates are kept on intentionally; do not strip them as a perf fix.

## Main constraints and how i got around them

Udon is slow and single-threaded, so per-voxel CPU work is unaffordable. Anything that touches all 4096 voxels of a chunk either runs as a GPU fragment shader pass or is time-sliced across many frames with hand-rolled cursor state machines (Udon has no coroutines).

The only GPU surface is VRChat's: `VRCGraphics.Blit` into RenderTextures, `VRCAsyncGPUReadback` for GPU-to-CPU transfers, `VRCShader.SetGlobal*` for `_Udon*`-prefixed globals. No compute shaders, no `Graphics.Blit` to null targets. One hard-won platform rule appears all over the code: globals set with `SetGlobalTexture` bind only at the fragment stage, so any texture a vertex shader needs must be material-bound and re-bound every frame (the block atlas ping-pongs between two RTs on every write).

Quest 3 must run all of it, which forces per-platform splits: GPU light-atlas capacity 2047 slots on PC vs 1023 on Quest, render distance 8 chunks vs 6, and the render distance silently clamps down at init until the lit column set fits the slot capacity. The fog end tracks the actual clamped render edge, not the nominal setting.

UdonSharp quirks are load-bearing. No multidimensional arrays (jagged or flattened only), no local methods, no try/catch. One VM bug matters everywhere: post-increment on a field used as an expression returns the wrong value, so every `x = f++` is split into a read plus an assignment. Check `udon_whitelisted*.txt` / `udon_blacklisted.txt` (generated by `Assets/Editor/UdonExposureExporter.cs`) before using any Unity API.

## World model and coordinates

| Property | Value |
|---|---|
| Chunk size | 16x16x16 blocks |
| World size | 32x8x32 chunks = 512x128x512 blocks |
| Chunk storage | `McWorld.chunks_1D`, flat array of 8192 `ChunkData` refs, lazily populated (null means "not generated yet", not "out of bounds") |
| Block storage per chunk | Tagged union in `ChunkData._chunkData`: boxed byte (homogeneous), raw `byte[4096]`, or per-column RLE `ushort[][]` |
| Block registry | 97 block IDs (0 Air through 96 Trapdoor) in `McBlockTypeManager`, editable via its custom inspector |
| Simulation rate | 20 TPS fixed timestep, decoupled from render framerate |

Coordinate handedness: Unity is left-handed, Minecraft is right-handed, so the world is stored X-mirrored relative to MC. The mapping is `x_mc = -x_vrcm - 1`, with y and z equal. Worldgen evaluates all noise and RNG in Minecraft coordinates and flips X when writing the column texture, so terrain is block-identical to a real b1.7.3 world at the same seed, just mirrored. The scene-view overlay (`SceneViewCameraPosition.cs`) shows both coordinate systems, and `McMovement.flipXAxis` applies the mirror when the player samples the world.

The scene YAML is authoritative for all world dimensions.

## Architecture

Four UdonSharp singletons do nearly all the work, all children of the `MinecraftGame` root object in the scene:

| Component | File | Role |
|---|---|---|
| `McWorld` | `Code/VoxelEngine/McWorld.cs` (~19.4k lines) + `McWorld.Scheduler.cs` | World container, frame governor, chunk scheduler, GPU backend, meshing, lighting, sky/fog/clouds, day/night |
| `McTerrainGenerator` | `Code/VoxelEngine/WorldGen/McTerrainGenerator.cs` (3 instances) | GPU column worldgen state machine |
| `McBlockTicker` | `Code/VoxelEngine/McBlockTicker.cs` | The b1.7.3 game-rules layer: fluids, sand, fire, growth, at 20 TPS |
| `ModifyTerrain` | `Code/VoxelEngine/ModifyTerrain.cs` | Player targeting (voxel DDA) and break/place |

`ChunkData` (plain C#, deliberately not an UdonSharpBehaviour) holds all per-chunk state in ~200 public fields. `McChunk` is an intentionally empty stub kept so the chunk prefab compiles.

### Frame loop and scheduler

Everything runs inside `McWorld.Update()` under a frame governor: one shared time envelope per Update (8 ms steady state, 24 ms during initial load), anchored at Update entry. The scheduler gets half of it (`schedulerBudgetShare 0.5`); the `ProcessActiveChunks` lanes consume the rest, with small work-gated floors (lighting 1.5 ms, near-mesh 1.5 ms, mesh decode 2 ms) so nothing starves on a blown frame.

The scheduler (`McWorld.Scheduler.cs`, the perf logs still print its section as "Coordinator:") runs 16 virtual workers, each a state machine: IDLE > DATA_GEN > LIGHTING > WAITING_FOR_MESH > MESHING. There is no threading; workers are just time-sliced cursors. Assignment priority: player-edit rebuilds first, then border healing, then new worldgen picks (radial, ring by ring, with a cache-affinity rule that drains the 8 sibling Y-chunks of an already-generated column before starting a new one), then deferred-mesh wakeups. Watchdogs force-release stuck meshes (300 frames) and orphaned generator slots.

Chunks stream Minecraft-style: data generates within render distance plus a 2-chunk margin, meshes build only inside the radial render distance, chunk GameObjects come from a pooled free list and are recycled when the player moves away.

### Terrain generation

A faithful port of b1.7.3's `ChunkProviderGenerate` + `MapGenCaves` + biome decorator, restructured as a GPU pipeline over a packed column RenderTexture (16 wide x 2048 tall = one 16x128x16 column, pixel `(x, y*16+z)`). Per column, in order:

1. Climate: temperature/rainfall/modifier simplex octaves plus a 64x64 biome lookup texture, read back async so the CPU knows biomes for decoration counts and tinting.
2. Density noise: the five Beta noise fields (16/16/8/10/16 octaves) as octave-accumulation blits over a 5x5x17 lattice, then `CombineFixed.shader` reproduces the exact Beta density formula, then trilinear interpolation fills stone/water/ice per block (`GpuColumnBaseFill`).
3. Surface: `GpuColumnSurfaceReplace` replays `replaceBlocksForBiome`'s top-down column state machine per fragment â€” the depth counter resets at every air block, so stone below terrain-noise overhangs gets its own grass/dirt surface exactly like vanilla, with top/filler mutations (sandstone exhaust, bare-stone, beach band) carried down the column. Per-column RNG draws come from the CPU; a separate highest-stone scan (`GpuColumnSurfaceInfo`) feeds decoration. The 16x16 sand/gravel/stone surface noises stay on CPU deliberately; the GPU version was provably wrong.
4. Caves: `GpuCaveCarve.shader` re-runs the entire `MapGenCaves` worm simulation per fragment over a 17x17 chunk neighborhood, replaying bit-exact `java.util.Random` streams on the GPU via `JavaRandomGPU.cginc` (48-bit LCG stored as `uint2`).
5. Decoration: the CPU replays the decoration PRNG of the 4 population-contributing chunks to produce tree/flower/grass candidates as texture pixels, and `GpuColumnTreeDecoration` / `GpuColumnDecoration` stamp them; cross-border tree trunks resolve against a pool of rendered neighbor "anchor" columns.
6. Readback: one 32 KB R8 `VRCAsyncGPUReadback` per column (~150 ms latency, the generator parks instead of polling), split into eight 4096-byte chunk slices. A GPU-resident mode that skips readback entirely exists but is off in the scene.

Parity rests on: a bit-exact C# `JavaRandom` (construction order of the noise generators is the seed contract), the GPU LCG port, permutation tables stored as 8-bit unorm textures (half floats corrupt them), CPU double-precision lattice coordinates uploaded as textures, and a deliberate replication of Beta's stale-Y gradient-cache bug. Every GPU stage has a CPU fallback with the same math (except caves and bedrock, which are GPU-only).

### GPU voxel backend and lighting

Nearby chunk data is mirrored into a GPU **block atlas**: one 16x256 tile per chunk slot (2047 slots PC / 1023 Quest, ~2896x3072 px ARGB32 on PC), plus a matching **light atlas**, both double-buffered. Three small CPU-authored indirection textures (slot lookup, slot meta, block props) let any shader sample any loaded chunk. Chunks inside the radial render distance are never evicted (evicting them was the historical "lighting goes black" bug).

Lighting is two shaders: a **seed** pass computes fresh sky columns and block emission for dirtied slots (the light-atlas alpha channel is the per-slot dirty bit), then a **propagate** pass is blitted up to 15 times per Update, each blit one parallel BFS step expanding light across all six directions and chunk borders. The atlas always stores noon lighting; day/night is a single global (`_UdonVRCM_SkylightSub`, 0-11) subtracted from the sky channel at sample time in the shaders, exactly like b1.7.3's `getBlockLightValue`. No geometry or relight work ever happens for time of day.

Smooth lighting is not baked: every terrain shader includes `MCTerrainGpuExactAo.cginc`, which does ~16 light-atlas taps per fragment and bilinearly blends 4 corner brightnesses through the Beta brightness curve. Samples that miss the atlas fall back to skylit-with-subtraction so streamed-out edges look right day and night.

GPU-to-CPU traffic is exclusively `VRCAsyncGPUReadback` through one shared completion handler routed by a FIFO of request kinds (face extract / rehydrate / batched), which relies on completions arriving in request order. Player edits go CPU-to-GPU as a single-pixel poke (`GpuSetBlockChain`) instead of a 16 KB tile re-upload.

### Rendering

Two mesh paths coexist:

The GPU shared-mesh path is the live steady state (`enableInstancedVoxelRender=1` in the scene). Every in-range chunk's MeshFilter is assigned the same 98,304-vertex cube-grid mesh; the vertex shader (`MCVoxelInstanced.cginc`) reads the block atlas, culls hidden faces via a 256x256 ShouldDraw LUT and degenerates them, and picks texture-array slices from a 256x1 face LUT whose alpha encodes render class (opaque / cutout / transparent / skip). Fully enclosed chunks get their renderers disabled outright. Never call `Clear()` on the shared mesh; pooled chunks may still reference it.

The CPU greedy mesher covers exact-geometry cases and the fallback: a resumable 6-axis greedy state machine emitting into pooled 12,288-vert buffers, used for chunks being edited that contain torches/crosses, and whenever the GPU backend is off. Sliced builds are byte-identical to atomic ones.

Three things always come from the CPU regardless of path: **fluid meshes** (fluids are excluded from every GPU render pass by design; `MCFluid.shader` renders them with opaque lava, a water depth-prime pass, then one blended color pass, replicating vanilla's translucent sorting without per-frame sorts), **decor meshes** (torches and cross-blocks, emitted in 48-cross slices), and **colliders**.

Environment systems are all analytic b1.7.3 ports: `MCSkyV2.shader` (dome fog, closed-form sunrise fan, sun/moon rotation about X, stars from a baked equirect generated by replaying `Random(10842L)`), clouds as real baked 3D box meshes (two 24-mesh tiles scrolled 0.03 blocks/tick and wrapped at the 3072-block period; a raymarched-plane version was tried and rejected), fog as a verbatim `EntityRenderer` state machine whose far plane tracks the actual clamped render edge, and water texture animation as a 20 Hz GPU port of `TextureWaterFX` spliced into the terrain and fluid shaders by texture-array slice index.

### Block simulation

`McBlockTicker` is the game-rules layer, ported behavior-by-behavior from the Java. A fixed-timestep accumulator gates simulation to 20 TPS (max 4 catch-up ticks per frame), because MC tick delays are tick counts: sand 3, water 5, lava 30, fire 40.

Each world tick: **scheduled ticks** pop from a 32,768-entry min-heap ordered by (due tick, insertion sequence), the same tie-break as MC's `NextTickListEntry`, deduplicated by a 65,536-slot hash set whose deletions must write tombstones (writing empty breaks probe chains and silently drops re-schedules; that was a real "water never recedes" bug). Then **random ticks**: 10 random voxels per chunk per tick, MC's density scaled to 16-cube chunks, with an independent time budget so fluid floods cannot starve them.

Ported behaviors include flowing fluids (the `getOptimalFlowDirections` flood fill at depth 4, lava's sticky 75% refusal to recede, water source duplication, lava+water hardening), falling sand/gravel (with a 16-slot visual entity pool running MC's 0.04 gravity), fire spread with the vanilla flammability table, grass spread/decay, leaves decay, saplings, crops, farmland hydration, ice/snow melt, mushrooms, cactus/reed/flower support checks.

Every mutation funnels through `McWorld._SetBlockGlobalWithMeta`, which mirrors `setBlockWithNotify`: write block (zeroing the metadata nibble, Beta parity), re-apply metadata before notifications, then notify all six neighbors and fire `onBlockAdded`. Two deliberate deviations from a naive port: flowing fluid does *not* reschedule on neighbor change (matching Beta; adding it makes recession ~13x too fast), and the flood fill treats not-yet-streamed chunks as flow-blocking because `GetBlock` reads them as air, which would fabricate phantom drops toward the streaming frontier.

### Player movement

`McMovement.cs` extends NUMovement and re-implements Beta's per-tick physics: 0.08 gravity, 0.91/0.8/0.5 air/water/lava drag, the exact ground-acceleration formula with the 0.16277136 constant, 0.42 jump motion, ladders, cobwebs, ice slipperiness, and fluid current push ported from `BlockFluid.velocityToAddToEntity`. It samples the world through a cached environment probe invalidated by `world.worldEditCounter`, and carries safety nets (push-out-of-solid, void-fall recovery) that only act on chunks that are actually meshed.

## Naming notes and dormant paths

Everything in the tree is either live or explicitly staged; there is no dead code. A few names still deserve a warning:

- The "instanced" render path mostly is not `DrawMeshInstanced`: it is one shared 98k-vert mesh on every chunk's own renderer with vertex-stage culling.
- `terrain_cutout.mat` uses `MCTerrain.shader` with a keyword; `terrain_trans.mat` points at `MCFluid.shader`.
- Fluid simulation is 100% CPU in `McBlockTicker._UpdateTick_FlowingFluid`, however GPU-flavored the surrounding systems look.

Dormant by scene toggle (implemented, off by default):

- Batched face readback, GPU-resident chunks (no CPU mirror), and resident-mesh-from-atlas with `GpuVoxelQuadDraw` â€” everything under `Gpu/Shaders/Resident/` and `Gpu/Materials/Resident/`. These are staged future work for the Quest readback pipeline; do not profile or reason about them as live.
- Game-layer stubs with no callers yet: `MinecraftGame`, `MusicManager`, `BaseEntity`, `McButtonArguments`. The block-outline visual is driven directly by `ModifyTerrain`.

## Configuration: scene vs code

Two sources of truth, split on purpose:

- Code-owned: most perf-tuning fields (frame budgets, tick budgets, lighting iterations) are marked `[System.NonSerialized]` with initializers, so the values in the source are what runs and the inspector cannot drift them.
- Scene-owned: plain public fields (world dimensions 32/8/32, render distances 8/6, `maxConcurrentWorkers 16`, feature toggles like `enableInstancedVoxelRender`) serialize from `Minecraft.unity`, and the scene value wins over the code default. The code-side defaults (a 4x2x4 world) are historical leftovers; trust the scene YAML.

`decode_blocks.py` at the repo root extracts the full 97-block registry (solidity, visibility, culling, shape, opacity, emission, UVs) straight out of the scene YAML without opening Unity.

## Debugging and tooling

- Perf summaries: with `LOGGING` defined, the console prints a multi-section performance report every 300 frames (Update lanes, scheduler/"Coordinator", terrain gen state histogram, GPU worldgen, block ticker dispatch counts, mesh building, GPU backend blit/upload totals). Most fixes in this repo started from one of these sections.
- Fluid debugging: `McBlockTicker.debugFluidFlow` logs every horizontal flow decision with per-direction neighbor/below/cost/optimal. On an anomalous all-directions-1000 flood-fill result it additionally dumps a `[FluidFlood1000]` block: the 7x7x2 neighborhood as the tick system's cache saw it versus direct world reads, plus per-cell readiness.
- Editor tools (`Tools > VRCMinecraft > ...`): cloud mesh baker, star field baker, fire texture generator, the seed tester window, and the Udon exposure-list exporter. Baked outputs are committed; rebake only when sources or parity logic change.
- MCPForUnity: an editor bridge (console reading, script validation, in-editor C# eval) used with AI assistance to speed up boilerplate and iteration.
- Sky probing: shader visuals can be verified headlessly by rendering temp cameras from editor code and reading the PNGs; this caught a sun/moon 180-degree swap that playtesting missed.

## Parity reference

The deobfuscated Minecraft Beta 1.7.3 source (RetroMCP) lives outside the repo at `D:/vrchat_projects/VRCMinecraft_Beta173_Deobf/RetroMCP`; only the `*.java` files are the reference. Code comments cite exact Java classes and line numbers (`BlockFlowing.updateTick`, `EntityRenderer` fog, `renderSky`, `TextureWaterFX`, `MapGenCaves`). When behavior is in doubt, the Java is the spec, including its bugs: the stale-Y gradient cache in terrain noise and the metadata-zeroing order in `Chunk.setBlockID` are reproduced deliberately.

Two known simplifications are documented tolerances, not oversights: leaf decay uses a 9x9x9 log scan instead of Java's ray-limited BFS, and crop growth rate approximates the full neighbor formula with hydration only.

## Known issues

- A hard-won UdonSharp rule, learned from a long fluid-flow divergence hunt: any method that calls itself must carry `[RecursiveMethod]`, or UdonSharp silently corrupts the caller's locals across the recursive call. `_CalculateFlowCost` is the one live recursive method and carries the attribute; the `debugFluidFlow` + `[FluidFlood1000]` diagnostics that caught it remain available.
- Sound, music, particles-for-everything, entities, and gamemodes are scaffolding-only so far.
- The GPU-resident chunk mode (chunks with no CPU mirror at all) is implemented but disabled in the scene pending the readback pipeline being Quest-3-proven.
